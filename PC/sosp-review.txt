Paper summary
-------------
This paper looks at how we can use CXL shared memory pools as an enabling
technology for single system image (SSI) operating systems.  The paper
describes a design that takes microkernel structures and splits them into
private and shared regions---placing the private ones in individual machines and
the shared ones in a CXL memory pool.  The authors show experimentally that the
system scales well on a setup that emulates a memory pool by placing each
virtual host on a socket of a multisocket host and using a CXL memory expander.

Strengths
---------
+ The vision is interesting and the use of CXL memory pools to support SSI is an
  interesting idea.

+ The system design itself is sensible---the state partitioning approach is
  sensible.

Weaknesses
----------
- The paper is missing an experimental comparison to any DSM-based or other
  SSI-supporting systems from the past.

- The paper lacks a complete discussion of its CXL requirements---what
  latency/bandwidth does the system require for reasonable performance?  What
  are its requirements in terms of the coherency and memory consistency model
  that it will require from the pool?

- The paper lacks a detailed discussion of the differences between the work and
  similar designs from the past such as the multikernel and LegoOS.

Comments for authors
--------------------
Thanks for submitting your work to SOSP 2025!

I like this project.  SSIs are a fascinating subject with decades of work.
They've long been an elusive vision of what could be and I commend you for
building a sensible design that aims to provide it.  The idea of state
partitioning is straightforward, but I think that is a strength of the paper
rather than a weakness.  The idea of integrating the design into a microkernel
is straightforward, especially since TreeSLS [SOSP 23] had a similar line of
reasoning only two years ago, but is also well considered.

I do have a few issues with the paper.  The first, and probably most important,
is that I think the design section is really missing a detailed discussion of
how the multikernel design [SOSP '09 ] and LegoOS [OSDI '18] differ.  Both
systems discuss similar ideas to the ones here, similar enough that it is a bit
difficult to track the precise differences beyond just "P3OS uses CXL".

The second issue is the evaluation.  The eval only compares P3OS with Linux,
there's no discussion of existing SSI or distributed shared memory systems.
There's decades of work there, surely something could run in your system to make
a comparison?  The current evaluation looks like what we might expect if P3OS
were the first ever SSI operating system; it isn't.

Finally, I think the paper could do a better job of establishing its CXL
requirements.  This could be in terms of rough performance assumptions, but I
think semantics are equally important.  In particular, the paper isn't totally
clear on what assumptions it makes about the memory consistency model of the
underlying hardware.  I don't see this as a showstopping limitation of the work,
but I think that the assumptions could be made more clear in the paper
(esp. since hardware companies are still designing these systems and might
benefit from your insights).

Paper summary
-------------
This paper presents P3OS, an OS-level single system image design. SSIs have not seen practical adoption due to message passing overheads. This paper proposes to use the shared CXL memory for synchronization instead. The challenge in using CXL is its limited bandwidth compared to DRAM and also that some CPUs can fail while others can continue to work (unlike original SSI design where the entire machine fails or it doesn’t). P3OS puts only shared state on CXL and keeps private state in local DRAM. Further, P3OS uses a microkernel design which is better suited for using the shared memory in a minimal way. This design is called the state-partitioned microkernel. The authors implement P3OS and show that it improves performance over the message-passing baseline under different applications.

Strengths
---------
+ Important problem: rethinking SSI in the context of shared CXL memory is interesting.
+ Realization that microkernels better fit the rack-scale computer is nice.
+ The authors implement P3OS and unmodified single-machine applications can run on P3OS, which is impressive. 
+ Solid evaluation.

Weaknesses
----------
- Although the problem is new, the individual techniques are not very novel. Most design aspect/techniques in P3OS are well-known. 
- Certain claims about SSIs and that current OSes are not aware of the DRAM/CXL hierarchy are questionable.

Comments for authors
--------------------
Thank you for submitting your to SOSP 25! I liked reading this paper. It presents an important problem and proposes to rethink SSI in the context of CXL shared memory, which seems very interesting to me. The idea of state partitioning and identifying that microkernels are a better fit for the CXL-based SSI are interesting. The design to allow thread and process migration are neat as well. The authors have implemented P3OS and experimented with real CXL memory. The evaluation is solid, exploring several use cases of P3OS. Overall, this was a good paper with nice motivation, good design principles, solid engineering and evaluation.   

If not for the below, I would have given stronger support for this paper. First, although the problem is new and the overall design makes sense, the individual design techniques are not novel. For example, it is obvious that only minimal state that is required across machines must be kept on CXL. Similarly, the idea of logging and recoverable locks has already been explored for the same problem of tolerating partial failures in the CXL-memory rack (e.g., Lupin). Similarly, the steps for thread/process migration are fairly straightforward given the basic design. Overall, I felt that the paper could benefit from explaining the novelty of the techniques more clearly. 

Second, I am not sure I agree with some of the claims about SSI. The paper states that OS-level SSIs are more efficient than application-level SSIs. There are many problems here. First, the paper subtly equates DSM systems to application-level SSIs. This is not necessarily true. A DSM implementation could very well be within the OS. The paper also argues that hypervisor-level SSIs are less effective than OS-level SSIs because OS has more context. By this logic, shouldn’t the application-level SSI have even more semantics and context and thus achieve better efficiency/performance?

Paper summary
-------------
P3OS is a single-system-image microkernel aimed at running on rack of
machines all connected via CXL to a switch and shared RAM. The kernel
supports applications with threads spread over many machines; IPC;
thread migration; system service processes spread over all machines;
it minimizes shared kernel data structures (keeping most in private
per-machine RAM); and performs fault-tolerant update of shared kernel
data structures via logging. The target applications are unmodified
threaded parallel programs, whose threads P3OS transparently runs
across multiple machines with CXL shared memory.

Strengths
---------
It's good to see work aimed at making it easier to program big compute
clusters, and specifically at exploiting the intriguing possibility of
rack-scale shared RAM via CXL. P3OS wrestles with good problems:
juggling between fast/local and slow/remote memory, and coping with
partial failure. It makes good use of shared fault-tolerant memory to
recover from partial failures, for example with a Frangipani-like
locking and logging scheme.

Weaknesses
----------
The paper would be stronger if it explained why a single-system-image
O/S is a compelling goal. Particularly given that the idea has been
around for decades but has never caught on.

DSM to support unmodified shared-memory threaded applications has run
up against two problems in the past: partial failure, and the
performance gap between local and remote memory. The paper would
benefit from a stronger story for why CXL and P3OS overcome those
problems. After all, remote CXL memory will probably have something
like 5x higher latency than local RAM, and P3OS appears to leave
recovery of applications from partial failure up to the applications.

The experimental setup (a single machine) is far enough away from the
target environment (a rack with fault-tolerant CXL switches and
fault-tolerant cache-coherent shared RAM) that it's hard to know what
the lessons are.

Comments for authors
--------------------
Given that the end goal is to help big parallel applications run on
multiple machines with shared CXL memory, it's a little odd that most
of the paper's ideas are at the O/S level. Perhaps a
single-system-image O/S is the real key to enabling big parallel
applications -- but this is a hypothesis that needs to be explicitly
stated and supported, particularly given the long history of this
idea. In many areas, e.g. recovering from partial failure and
minimizing shared state, P3OS solves problems for itself, but doesn't
do much to help applications. The design rationale might be more
convincing if it started from the question "what would it take to make
it easy to write big parallel programs with CXL shared memory" rather
than "how can we make an SSI O/S on CXL?"

An alternate approach might be to use an existing single-machine O/S
like Linux and manage the shared CXL memory mostly at application
level. It would be good if the paper made clear why P3OS's approach is
better. The paper says (in 2.1) that P3OS is preferable to
application-level SSI because P3OS is fully transparent to
applications, but this is debatable since applications need to be
modified to deal with partial failures, and would also probably end up
being modified to cope with CXL's factor-of-five higher latency than
local RAM.

It would be good to make the assumed CXL setup more clear. Is the idea
that CXL allows hosts to directly read/write each other's RAM? Or are
all the hosts connected to a CXL switch, which allows them to
read/write special CXL RAM that's also attached to the switch?

The paper assumes the existence of rack-scale fault-tolerant
(replicated?) CXL switches with cache-coherent fault-tolerant
(replicated?) RAM, with access latency of a few hundred nanoseconds.
The paper has a few citations for this, but they seem pretty
speculative. The closer the paper can get to naming available
hardware, the better.

Much of the stated attraction of P3OS is that it can run existing
single-machine threaded parallel applications on multiple machines
without modifying the applications. But the paper also assumes that
applications can recover from partial failures. That seems like it
would require substantial modifications to applications, or entirely
different approaches than shared-memory threads (e.g. architectures
that handle faults well, such as MapReduce or Spark).

Do the machines need to communicate directly with each other,
similarly to IPI, e.g. when modifying page tables? Or is all
interaction via the CXL shared memory, so that each machine has to
poll for messages from other machines?

The statement at the end of 6.2.2 that P3OS doesn't have a plan for
recovering the private state of failed system services is troubling.

The paper often mentions microkernels in contexts where it's clear the
authors are talking about some particular microkernel (e.g. when
mentioning capability tree-based object management). It would be
helpful to say from the start which microkernel is under discussion.

What CXL hardware does the evaluation use? Why do the experiments use
CXL at all? It seems that it is a kind of CXL that has nothing to do
with sharing memory between machines.

It's too bad that the experiments are on a single machine, rather than
on multiple machines attached to a CXL switch and shared RAM. I don't
think I really understand the relationship between the presented
measurements and what one would expect on a real deployment.

Would it be possible to compare application performance with P3OS
against applications that use CXL shared memory with other approaches?
In order to show that P3OS is a better way to use CXL than other ways?

Paper summary
-------------
This paper’s focus is enabling fault-tolerant SSI-like distributed OSes on racks with disaggregated shared memory. The paper argues that the emergence of shared memory makes SSI feasible, since it alleviates the need to serialize data when going between machines. However, it also suggests that current monolithic, which do not partition local state (e.g., state needed by a single core or a single NUMA node) and shared state cannot easily be adapted to this setting. Therefore, the paper designs a new microkernel where state shared across servers is stored in the shared memory blade, and other state is stored in server local memory. The paper argues that capability based access control makes it easy to perform the same partitioning for individual processes too.

Strengths
---------
Proposes an interesting approach to building a single operating system that spans a disaggregated rack.

Weaknesses
----------
* Many of the hard/interesting problems are punted to prior work or to applications. 
* No motivation for why this is the right way to use a rack scale machine.

Comments for authors
--------------------
I found the premise of this paper intriguing, and I was really interested in seeing the technical details. Unfortunately, the paper as written makes it hard to really get a deep understanding: much of the paper (e.g., the assumption of a capability tree for access to system services, the implementation detail, etc.) seems to build on a **specific microkernel** (not all of them use capabilities, etc.) but what microkernel it assumes is never mentioned. 

But even setting aside that problem, partitioning state seems like a reasonable idea. However the challenging bit once you partition (IMO) comes down to dealing with synchronization primitives (e.g., locks left in an awkward state due to failures), state consistency, and the loss of state in disaggregated memory. But the paper punts on all of these:

* Section 6.2.1, relies on prior work for kernel locks, and an undescribed WAL for state consistency for synchronization and state consistency in the kernel. It punts these problems to application developers, and I am not entirely convinced that this is easier for them to solve. 

* Section 6.1, puts memory blade failures out of scope, despite the introduction mentioning them as a challenge. 

So overall the techniques are interesting but seem incomplete. 

But there is a bigger issue with this paper in my opinion: I am not sure why one should use a rack scale disaggregated setup in this way, rather than the more traditional treatment of multiple servers connected to a shared memory pool that can be used to increase memory capacity and for communication. The paper never really makes a case for this, and seems premised on the idea that SSI is a good design. But is it? For what applications? 

As it stands, I think the paper describes an interesting technical riddle, but the solution it proposes does not seem particularly interesting, and I  am not sure whether solving the riddle is important or useful.

Paper summary
-------------
The paper describes a single system image microkernel that uses CXL memory.

Strengths
---------
+ Timely problem and technically viable approach.
+ Wide variety of benchmark applications.

Weaknesses
----------
- The need for a single system image (SSI) microkernel to support CXL is not technically justified.
- The technical solutions are either vague or a subset of previous work.
- System does not obtain the partial failure tolerance of two previous works.
- Evaluation system has CXL attached to one socket, yet NUMA effects are not discussed.

Comments for authors
--------------------
Thank you for your submission.  Like many researchers, I think CXL memory holds promise for operating systems and software systems.  I think there is a version of this paper that discusses a SSI microkernel and its support for CXL that would be acceptable for SOSP, but unfortunately, I do not think this submission reaches that level.

I would say this submission has three goals: (1) convince the reader that a SSI microkernel is the right system support for CXL; (2) efficiency, as the introduction states, simply placing all OS data in CXL will slow down the system; (3) fault tolerance, in particular the problem of partial failures, which is fairly novel in the CXL context.

The submission makes no technical or quantitative case for (1), it simply asserts it the conclusion as fact.  This is puzzling because [66] and [69] are both very relevant related work, and noted as such.  Both of them built on a traditional monolithic operating system structure, and both obtain superior fault tolerance to the proposed system, so where is the need and benefit of the microkernel?  The conclusion of the authors, summarized in Figure 2 is to only place shared data structures in CXL memory, which is also the conclusion of [66] and [69], and which can be achieved in the context of a normal operating system, which only has some of its state stored in CXL memory.

Previous work has established that there are two alternatives for tolerating partial failures, detectable lock-free data structures [see Friedman PPoPP 18, and Memento PLDI 23], and recoverable locks.  The submission clearly chooses recoverable locks.  That is great, and if the submission detailed its solution to the many problems of recoverable locks, that could be a significant contribution.  How many locks are there in the kernel (or system services)?  How deeply do they nest?  How much space is required for the log?  How much data is logged per lock acquire?  How is log space reclaimed?  What is the performance overhead of logging for recoverable locks?  How do you do atomic lock ownership transfer?  How do you track lock ownership?  How do you avoid problems if a thread is thought dead, but recovers?  How do you detect dead threads?  If a timeout is involved, how can you set it so that latency is low, but there is no danger of a false positive?  When threads recover do they regain the same identity or are they given a new identity?

The submission does not provide any meaningful detail on any of these issues.  We are told things like 4.1.2 says there is a cross-machine shared queue.  Section 5 has details, but these details are along the lines that stack memory is kept thread local.  That is fine, but obvious.  There is private replicated state, but we never learn what kernel or system services use this feature or how.  6.2.1 says the system uses recoverable ticket locks and journaling, but no details on the logging overheads or the number of locks or their nesting or the ownership or the recovery strategy is given.  The submission does not contain enough detail to substantiate how it achieves efficiency and correctness.

The submission does not identify its base microkernel, compounding the difficulty of understanding its contribution.

Fault tolerance is the biggest disappointment in the work, given how the main contribution of [66] and [69] is partial failure tolerance (achieved by reference counting and detectable data structures, respectively).  After many pages of equivocation, 5.2 admits that for user programs, the system places the stack in private memory and the heap in CXL.  This strategy is worse than both [66] and [69] and shown to be problematic for performance for many applications in Pond ASPLOS 23.  It assumes that applications already have their own recovery mechanisms, like a database.  Very few applications are recoverable, and POS provides no system support for these applications.

The submission touts the fact that it uses unmodified binaries, which does allow it to run an impressive array of benchmark programs.  Unfortunately, existing prototypes can already run a variety of benchmark programs.  For example, Pond [ASPLOS 23] evaluates many applications and shows that many suffer performance problems if their entire heap were placed in CXL memory.  The submission evaluates single applications, but in any real system if most of the applications kept their heap in CXL, the bandwidth and latency limitations would surely limit system performance.

The submission's microkernel has system services.  Designing system services for CXL is potentially interesting and also potentially difficult.  Once again, the submission is vague on details.  Figure 3 shows that the file system tracks what machine manages which file.  That is fine, but what happens when a machine crashes?  Is file data cached locally or in CXL?  Buffered writes are not allowed in earlier versions of NFS to prevent problems due to crashes, what does POS do?  How does it determine which host controls which file?  The submission mentioned obvious mount points like m1 for machine 1, but does every valid path start with a machine identifier?  What machine is responsible for /dev?  Is it always the same machine number?  How efficient or inefficient is accessing a local file via a remote file via RPC?  These issues are resolvable and the answer to them might form the technical backbone of a more detailed revision of the submission.

I did not follow the importance of the capability tree in 5.1.  What is an object?  Is a linked list an object, or is each node in a linked list an object?  Are objects only OS-level resources like memory pages?

The experiment in 9.4.3 nicely shows the advantage of disaggregated memory and how it prevents the stranding of resources.  This motivation stretches back to at least LegOS in OSDI 18 and it is explicitly mentioned and measured in Pond ASPLOS 23.

Response by Author [Fangnuo Wu <wufangnuo@sjtu.edu.cn>] (1257 words)
---------------------------------------------------------------------------
**We first address common questions, followed by responses to individual reviewers.**

**Q1. Why is SSI OS the right way to support applications in CXL rack?**

SSI-OS is compelling for running single-machine applications in CXL-rack for several reasons.

1. SSI-OS supports running zero-changed single-machine legacy applications. In contrast,  DSM requires non-trivial modifications: developers should identify shared variables, modify their allocation to CXL, and recognize that certain variables (at different locations of different processes’ address space) refer to the same variable. Lupin[69]/CXL-SHM[66] reported they use several days to port a simplified MapReduce. 

2. It is easier to develop applications via shared memory provided by SSI-OS; message passing over CXL forces programmers to use explicit message-passing, which is harder to develop and sometimes less efficient due to communication overhead.

3. SSI-OS enables resource disaggregation. Applications can access remote resources with legacy interfaces. Also, SSI-OS provides more ability than Pond/LegoOS: it enables thread-level resource scheduling across the rack to utilize stranded CPUs thus improving rack‘s overall performance. 

**Q2. How does P3OS help applications recover from partial failures?**

P3OS does not provide recovery of applications. P3OS handles partial failures from the OS's perspective so that (1)processes influenced by machine failures will be terminated as if they are killed on a single machine, and their resources are reclaimed; (2)system services and other processes will not be blocked/crashed by failures.

Recovering applications is beyond our scope, since single-machine applications typically can tolerate machine failures by themselves (e.g., databases use WAL; stateless computation tasks simply restart or recover from their persisted intermediate results). If necessary, P3OS can offer a checkpoint/restore service or a recoverable in-memory data structures SDK for applications to use.

Notably, P3OS provides the same partial failure guarantee as CXL-SHM[66], which supports non-blocking memory allocation but not application recovery. Another prior work, Lupin[69], provides application recovery by writing applications (MapReduce/PARSEC) with recoverable data structures. In addition to supporting unmodified single-machine applications, P3OS can adopt Lupin’s approach (which sacrifices transparency) to support application recovery.

**Q3. What are the novel design aspects of P3OS compared to prior OSes like MultiKernel and LegoOS?**

P3OS has two novel design aspects: 

1.New kernel architecture. Multikernel/LegoOS employ partitioned kernels communicating via message passing and do not support shared memory between machines. P3OS leverages inter-machine shared memory, and places part of its kernel state on CXL to utilize the shared memory semantic in CXL. Our designs are based on shared-memory-based kernel architecture.

2.Novel partial failure handling (of P3OS itself). Multikernel does not consider partial failures of a single kernel. LegoOS only solves memory component failure (by replication). Instead, P3OS handles partial failure problems caused by inconsistent memory state on CXL.

**Q4. Would the findings from experiments on a single machine carry forward to real CXL 3.0 rack hardware?**

Yes. Our setup with CXL 1.1 device has similar performance to the reported real CXL 3.0 platforms. Our device has 3.6x latency compared to local DRAM (Table1). Previously reported FPGA-based CXL-devices (Alibaba's work @SIGMOD25) and ASIC-based CXL-devices (according to [58]) have 3.7x and ~3.0x latency respectively. Such latency includes the latency of CXL switches.

Rack-scale cache-coherence might introduce extra latency, less than 30% in our cases. CXL 3.0 will cause 1-2x higher latency for the cache miss due to cross-host cacheline invalidation (according to [58]). Applications we tested have 5-15% cache miss rate, thus the overhead will be <15%*2=30%.


**Q5. How would keeping applications' heap on CXL impact performance?**

Placing heap on CXL introduces acceptable additional overhead, since P3OS does not allocate all applications to CXL memory. In our two target scenarios:

1.Scaling applications (Figure 10): CXL incurs overhead on a single machine (20%-80% in Figure10), but the overall performance improves when scaling to 8 machines (by 2-6x).

2. Utilizing Stranded CPU (Figure 12): P3OS schedules applications with small heaps to leverage CPU on multiple machines, where the performance impact remains minimal (1%-17%).
 
**Q6. Can you identify the microkernel that was the basis for your implementation?**

P3OS is based on ChCore, the open-sourced kernel of TreeSLS. Our design can also be applied to other microkernels with the capability trees (such as seL4).
 
**Q7. Can you provide details on your recoverable lock design, the resources used by these locks at runtime, and the recoverability experiments you ran to validate the correctness of the design? Did you consider overlapping failures?**

Our recoverable lock design is similar to Lupin. The recoverable lock set machine ID when acquiring a lock with CAS. We use a global heartbeat-based detector to detect machine failures, and once a machine is considered to be dead, the detector will clear all locks held by it.

**Overhead**:

1. Resources used by recoverable locks are limited. Components using recoverable locks include CXL memory allocator, cross-machine IPC/scheduling queue. They use only a few dozen locks. System services instances are running on each single machine and do not use recoverable locks.

2. The runtime overhead of recoverable lock & log is minimal (<1%) in above components. First, acquiring/releasing the locks is simply compare-and-swapping machine-ID. Second, our lightweight logs, which track the latest in-flight operations (e.g., enqueue, malloc) to ensure their atomicity, merely write an entry to the CXL memory, which introduces minimal overhead (<1%). 

3. Unreleased locks before its recovery will not block the running of P3OS since: (1) CXL memory allocators use multiple buddy/slab pools, so allocators can find one usable pool to allocate unless all pools are locked by the failed machine. (2) Scheduling or IPC to failed machines might be blocked by the unreleased locks. Instead, the scheduling will cancel the migration to the failed machine, and IPC to system services must wait for the recovery of that system services (5.5ms for file system in Figure 13).

**To test correctness**, we crash the VM after acquiring the lock and make sure other machines still function well.

**P3OS considers overlapping failures**. When multiple machines fail together, we just clear their locks one by one.

**Reviewer A and C:**

**Q1: Clarify CXL setup.**

As Section 2.1 describes, we have several hosts connected to a CXL 3.0 switch with CXL RAM attached to the switch. CXL memory requires global cache coherence support.
 
**Reviewer C:**

**Q1: Do the machines communicate directly with each other? Is all interaction via the CXL shared memory or IPI?**

Yes, but through neither IPI nor polling. P3OS uses scheduler-based IPC: When machine A needs to communicate with machine B, A first prepares a message and then wakes up a SHADOW thread by moving it to B’s scheduling run-queue. B then processes the message. If the CXL hardware provides IPI support, we can leverage it to further optimize communication.

**Reviewer E:**

**Q1: Several questions about FS.**

1. When a machine crashes, FS instances on this machine will recover on another machine (Figure 13).
2.File data is cached locally, so buffered writes are not persisted, and we only recover persisted files. Making cached data persisted can leverage methods in Ananke (FAST25).
3.To determine which host controls which file, local FS identify which mount point the file belongs to, and send requests to the responsible FS instance according to a list maintained in a table on CXL (Figure3(b)).
4.Not every valid path starts with a machine identifier. Normal path can be valid when it is registered as a valid mount point.
5.One machine can take responsibility for /dev/ (not always the same machine), and when this machine fails, another machine can take responsibility.
6. Accessing remote 4KB files has extra ~600ns compared to accessing local files (Table2).
 
**Q2: What is an object in the capability tree?**
OS-level resources like memory pages, IPC buffer, etc.
